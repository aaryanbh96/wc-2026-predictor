# World Cup 2026 Predictor

A self-contained web app that rates all 48 teams, predicts each match beforehand,
locks the prediction into a ledger, then grades and learns from it once the result
lands. Ships pre-trained on the real round-1 group results. Optionally auto-updates
from football-data.org every hour via a GitHub Action.

## Files

- `index.html` — the whole app. No build step, no dependencies. Open it locally or host it.
- `.github/workflows/update-results.yml` — hourly Action that fetches finished results.
- `scripts/fetch_results.py` — the script the Action runs; writes `results.json`.
- `results.json` — generated by the Action; the page reads it on load. (Not committed until the Action first runs.)

## Deploy on GitHub Pages

1. Create a repo and add these files.
2. Settings → Pages → Build from branch → `main` / root. Your app is live at `https://<you>.github.io/<repo>/`.

That alone gives you the full predictor with manual result entry. The auto-update is optional.

## Turn on hourly auto-updates

1. Get a free API token at https://www.football-data.org/ (register, copy your token).
2. In the repo: Settings → Secrets and variables → Actions → New repository secret.
   Name it `FOOTBALL_DATA_TOKEN`, paste the token, save.
3. The Action runs hourly on its own. To test immediately: Actions tab →
   "Update World Cup results" → Run workflow.

When it runs, it writes `results.json` to the repo. The page fetches that file on
load (and via the **Sync now** button), settles any finished match it hasn't seen,
grades any locked prediction for it, and learns. Matches already baked into the seed
are recognized and skipped, so nothing is double-counted.

### Notes & limits

- **Free tier:** 10 requests/minute. The Action makes one call per hour — far under the limit.
- **Score lag:** finished scores can post with some delay on the free tier. The manual
  "Log result" tab is always available as a fallback.
- **Name mismatches:** football-data.org may spell teams differently (e.g. "Iran" vs
  "IR Iran"). `scripts/fetch_results.py` has a `NAME_MAP` for the known cases; if the
  Action logs an "unmatched" warning, add the mapping there.
- **Storage:** all state (ratings, ledger, learning history) lives in your browser's
  localStorage — it's per-device, not shared across visitors.

## How it works

The model is Elo-style: each team has a rating, and `P(A wins) = 1/(1+10^((Rb-Ra)/400))`.
After each result it nudges ratings toward the truth, scaled by margin and surprise —
that's the online-learning loop. The pre-match ledger records each prediction *before*
the match so the headline accuracy reflects genuine ahead-of-time calls, not hindsight.

It only sees ratings — not injuries, rotation, or tactics — so treat it as a transparent
learner, not a betting edge.
